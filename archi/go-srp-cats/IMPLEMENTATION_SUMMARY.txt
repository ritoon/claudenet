KEY FEATURES
================================================================================

STRICT SRP IMPLEMENTATION
- Each file/component has exactly ONE responsibility
- Each component has ONLY ONE reason to change
- Zero mixing of concerns across files
- Complete isolation of concerns

FULL CRUD OPERATIONS
- Create: POST /cats (201 Created)
- Read One: GET /cats/{id} (200 OK)
- Read All: GET /cats (200 OK)
- Update: PUT /cats/{id} (200 OK)
- Delete: DELETE /cats/{id} (204 No Content)

CAT ATTRIBUTES
- ID (string) - automatically generated
- Name (string) - required, max 100
- Breed (string) - required, max 100
- Color (string) - required, max 100
- Age (int) - required, 0-50 range

VALIDATION RULES
- Name: Required, non-empty, max 100 characters
- Breed: Required, non-empty, max 100 characters
- Color: Required, non-empty, max 100 characters
- Age: Integer between 0 and 50

IN-MEMORY STORAGE
- Thread-safe using sync.RWMutex
- Generic key-value store (easily replaceable)
- Methods: Get, Set, Delete, GetAll, Exists

SECURE ID GENERATION
- Uses crypto/rand for cryptographic randomness
- 16-byte random IDs as hexadecimal strings
- Isolation means swapping for UUID/Ulid is one-file change

NO EXTERNAL DEPENDENCIES
- Uses only Go standard library
- net/http, encoding/json, sync, crypto/rand, fmt, log
- No frameworks, no ORMs, no external packages
- Minimal Go version: 1.21

================================================================================
SRP ENFORCEMENT EXAMPLES
================================================================================

EXAMPLE 1: Change Validation Rule
Scenario: Age limit changes from 50 to 40

Before: Edit would affect multiple files potentially
With SRP: Only validation/cat_validator.go changes
  Line to change: if cat.Age < 0 || cat.Age > 40 { ... }
  Other files: COMPLETELY UNTOUCHED

EXAMPLE 2: Switch Storage Backend
Scenario: Move from in-memory to PostgreSQL

Before: Would require changes throughout codebase
With SRP: Only storage/memory_store.go and possibly repository/cat_repository.go
  Handlers: UNCHANGED
  Service: UNCHANGED
  Validation: UNCHANGED
  Serialization: UNCHANGED
  Routing: UNCHANGED
  Entity: UNCHANGED
  Errors: UNCHANGED

EXAMPLE 3: Add New Endpoint
Scenario: Add GET /cats/breed/{breed}

With SRP:
  1. Create handler/get_cats_by_breed.go (NEW FILE)
  2. Add GetByBreed method to repository/cat_repository.go
  3. Add GetCatsByBreed method to service/cat_service.go
  4. Add route to routing/router.go
  
Unchanged files: 11 other files don't need modification

EXAMPLE 4: Change Output Format (JSON to XML)
Scenario: API clients want XML instead of JSON

Before: Would touch handlers, service, everywhere
With SRP: Only serialization/response.go changes
  - Change json.Marshal to xml.Marshal
  - Update CatResponse struct tags
  Handlers: UNCHANGED
  Service: UNCHANGED
  All others: UNCHANGED

================================================================================
DESIGN PATTERNS USED
================================================================================

1. DEPENDENCY INJECTION
   - main.go creates all instances
   - Wires dependencies explicitly
   - No global state
   - Testable (easy to mock)

2. REPOSITORY PATTERN
   - cat_repository.go abstracts storage
   - Service talks to repository, not storage directly
   - Can swap storage implementation

3. SERVICE LAYER PATTERN
   - cat_service.go orchestrates operations
   - Contains business logic coordination
   - No HTTP concerns
   - No storage concerns

4. HANDLER PATTERN
   - Each handler = one endpoint
   - Consistent: Decode → Service → Encode
   - No duplication
   - Easy to test

5. DTO PATTERN
   - Request/Response DTOs in serialization/
   - Separates API format from domain entities
   - Can change API contract without changing domain

6. ERROR HANDLING
   - Custom domain error types
   - Type assertion for error handling
   - Proper HTTP status codes

TESTING CAPABILITIES
================================================================================

The architecture enables unit testing in isolation:

VALIDATOR TESTING (NO HTTP, NO DATABASE)
- Create CatValidator instance
- Call ValidateCreate/ValidateUpdate
- Assert errors
- No mocks needed (pure logic)

SERVICE TESTING (NO HTTP)
- Mock CatRepository
- Mock CatValidator  
- Mock IDGenerator
- Call service methods
- Assert results without HTTP layer

HANDLER TESTING (NO SERVICE IMPLEMENTATION)
- Mock CatService
- Send HTTP requests to handler
- Assert status codes and response bodies

STORAGE TESTING (ISOLATED)
- Create MemoryStore
- Test Get/Set/Delete/GetAll
- No domain logic, no HTTP
- Could test PostgresStore same way

REPOSITORY TESTING
- Mock MemoryStore
- Test CatRepository methods
- Assert error handling

================================================================================
EXTENSIBILITY PATHS
================================================================================

1. NEW VALIDATION RULE
   Modify: validation/cat_validator.go
   Impact: Single file change

2. NEW STORAGE ENGINE
   Modify: storage/memory_store.go (or replace)
   Impact: 1-2 files maximum

3. NEW ENDPOINT
   Create: handler/new_operation.go
   Modify: repository/cat_repository.go (if needed)
            service/cat_service.go (if needed)
            routing/router.go
   Impact: Localized, no existing handlers affected

4. NEW OUTPUT FORMAT
   Modify: serialization/response.go
   Impact: Single file change

5. NEW INPUT FORMAT
   Modify: serialization/request.go
   Impact: Single file change

6. NEW DATA FIELD
   Modify: entity/cat.go
            serialization/request.go
            serialization/response.go
            validation/cat_validator.go
            service/cat_service.go
            handlers (parameter changes)
   Impact: Localized to affected layers only

7. DATABASE MIGRATION
   Modify: storage/ layer (new implementation)
           repository/ layer (if query logic changes)
   Impact: Business logic completely untouched

================================================================================
HTTP STATUS CODES RETURNED
================================================================================

201 Created
- Returned by POST /cats when cat created successfully
- Body: CatResponse with generated ID

200 OK
- Returned by GET /cats when all cats retrieved
- Returned by GET /cats/{id} when cat found
- Returned by PUT /cats/{id} when cat updated
- Body: CatResponse or []*CatResponse

204 No Content
- Returned by DELETE /cats/{id} when cat deleted successfully
- No body

400 Bad Request
- Returned when request body is invalid JSON
- Returned when validation fails (business rules)
- Body: ErrorResponse with error message

404 Not Found
- Returned by GET /cats/{id} when cat not found
- Returned by PUT /cats/{id} when cat not found
- Returned by DELETE /cats/{id} when cat not found
- Body: ErrorResponse with error message

405 Method Not Allowed
- Returned when HTTP method doesn't match endpoint
- Body: ErrorResponse with error message

500 Internal Server Error
- Returned when unexpected error occurs
- Body: ErrorResponse with generic message

================================================================================
SOLID PRINCIPLES COVERAGE
================================================================================

S - SINGLE RESPONSIBILITY PRINCIPLE (STRICTLY ENFORCED)
✓ Each file has exactly one responsibility
✓ Each file has only one reason to change
✓ This entire project is an example of perfect SRP

O - OPEN/CLOSED PRINCIPLE (DEMONSTRATED)
✓ Open for extension (new handlers, new validators)
✓ Closed for modification (don't modify handlers to change behavior)
✓ Example: Add new endpoint without modifying existing endpoints

L - LISKOV SUBSTITUTION PRINCIPLE (DEMONSTRATED)
✓ MemoryStore can be replaced by PostgresStore
✓ Contract maintained (same interface)
✓ Can't break existing code by swapping implementation

I - INTERFACE SEGREGATION PRINCIPLE (IMPLICIT)
✓ Service depends only on needed methods
✓ Handler depends only on service interface (conceptually)
✓ Focused, single-method contracts

D - DEPENDENCY INVERSION PRINCIPLE (DEMONSTRATED)
✓ High-level modules (handlers) don't depend on low-level (storage)
✓ Both depend on abstractions (repository interface concept)
✓ Dependencies injected from main.go
